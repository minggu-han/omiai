# 项目需求文档：AI 约会助攻顾问 (AI Dating Coach)

## 1. 项目概述

### 1.1 项目背景

当代年轻人在微信/社交软件上经常面临"不知道回什么""怕回错话"的社交焦虑。本项目利用 LangGraph 构建一个多智能体协作系统，帮助用户分析聊天语境、生成个性化回复，并通过**人工审核机制**确保回复符合用户真实人设。

### 1.2 项目目标

- 构建一个支持**人机协同（Human-in-the-loop）**的对话辅助系统
- 实现 **4个核心AI节点 + 1个人工中断节点** 的完整工作流
- 演示 LangGraph 的状态管理、条件分支、循环回流能力
- 最终输出：**可运行的命令行/Web交互原型**

### 1.3 核心价值

- **安全感**：所有AI生成的回复必须经过用户确认才能输出，避免"社死"
- **策略性**：不只给一个答案，而是给多种风格选项，让用户有选择权
- **可解释性**：AI会解释"为什么这么回"，帮助用户提升聊天能力（长期价值）

------

## 2. 用户角色与使用流程

### 2.1 用户角色

- **普通用户**：正在经历暧昧期/约会初期的单身人士
- **（后期可扩展）情感咨询师**：可以审查AI的建议质量

### 2.2 核心使用流程（端到端）

text

```
用户输入聊天记录 
    ↓
AI分析对方意图和情绪
    ↓
AI生成3种风格回复（幽默/温暖/推拉）
    ↓
⚠️ 系统中断，等待用户人工审核 ⚠️
    ↓
用户选择：接受 / 修改后接受 / 完全驳回
    ↓
（若驳回）→ 回到生成节点重新生成
（若接受/修改）→ AI润色最终版
    ↓
AI模拟对方看到回复后的3种可能反应
    ↓
输出最终回复 + 心理预期剧本
```

## 3. 详细功能需求

### 3.1 输入层需求

| 功能点               | 描述                                                         |
| :------------------- | :----------------------------------------------------------- |
| 聊天记录输入         | 支持粘贴纯文本聊天记录，格式要求：`对方：xxx` / `我：xxx`    |
| 用户人设设定（可选） | 用户可简述自己的性格（如"内向理工男""活泼搞笑女"），影响回复风格 |
| 目标意图（可选）     | 用户可选择本次聊天目标（"邀约见面""化解尴尬""推进关系"）     |

### 3.2 核心处理节点

#### Node 1: 意图分析节点 (Analyzer)

- **输入**：聊天记录（最近10-20条）

- **处理逻辑**：

  - 分析对方情绪倾向（热情/冷淡/试探/防御）
  - 识别对方的潜在意图（想结束对话/想延续话题/在分享情绪/在测试反应）
  - 评估当前对话的"危险系数"（是否需要紧急转移话题）

- **输出字段**：

  json

  ```
  {
    "emotion_score": 0.7,  // 0-1，越高越正向
    "intent": "分享日常",
    "risk_level": "低",
    "key_insight": "对方在主动找话题，建议积极回应并抛出新话题"
  }
  ```

  

#### Node 2: 策略生成节点 (Generator)

- **输入**：分析报告 + 用户人设 + 目标意图

- **处理逻辑**：生成3种截然不同风格的回复

  - **风格A - 幽默调侃型**：带表情包文字，轻松化解紧张
  - **风格B - 温暖共情型**：展示理解和关心
  - **风格C - 推拉高手型**：略带挑战/神秘感，制造吸引力

- **输出字段**：

  json

  ```
  {
    "candidates": [
      {
        "style": "幽默",
        "reply": "哈哈哈你这说的我差点以为你在监控我生活",
        "strategy_explain": "用夸张认同拉近距离，制造轻松氛围"
      },
      // ... 共3个
    ]
  }
  ```

  

#### Node 3: 👨‍💻 人工审核节点 (Human Reviewer) - **关键中断点**

- **输入**：3条候选回复 + 各自策略说明

- **交互方式**（命令行版）：

  text

  ```
  === 请审核以下回复 ===
  1. [幽默] xxx
  2. [温暖] xxx
  3. [推拉] xxx
  
  请选择操作：
  [A] 接受第X条  [B] 修改第X条  [C] 全部驳回重来  [D] 退出
  ```

  

- **输出**：用户的决策 + 修改内容（如有）

- **状态变更**：系统在此节点**完全阻塞**，等待用户输入后恢复

#### Node 4: 回复优化节点 (Refiner)

- **输入**：用户选中的候选 / 用户修改后的草稿
- **处理逻辑**：
  - 如果用户选了"接受"：仅做语法润色和口语化调整
  - 如果用户选了"修改"：根据用户的修改意见重写，保留用户的核心意思
  - 如果用户选了"驳回"：**不执行此节点**，直接跳回Generator
- **输出**：精修后的最终回复文本

#### Node 5: 模拟推演节点 (Simulator)

- **输入**：最终回复

- **处理逻辑**：模拟对方收到回复后的3种可能反应

  - 最佳情况（对方积极回应，开启新话题）
  - 一般情况（对方简单回复，不冷不热）
  - 最差情况（对方误解/冷漠，需要补救）

- **输出**：

  json

  ```
  {
    "best_case": "哈哈哈你可真会说话，那周末要不要出来？",
    "normal_case": "哈哈 好的吧",
    "worst_case": "你这话什么意思？",
    "fallback_suggestion": "如果对方回'你什么意思'，建议立刻道歉并解释是开玩笑"
  }
  ```

  

### 3.3 输出层需求

- 最终呈现给用户的完整报告包含：
  1. ✅ 优化后的最终回复（可直接复制）
  2. 📊 对方反应模拟剧本
  3. 💡 避坑建议（如果对方最差反应，该怎么挽回）

## 4. 技术架构需求

### 4.1 技术栈

| 组件         | 技术选型                                 | 说明                     |
| :----------- | :--------------------------------------- | :----------------------- |
| 框架         | **LangGraph**                            | 核心工作流编排           |
| 状态管理     | LangGraph State + MemorySaver            | 实现Checkpointer中断恢复 |
| LLM          | OpenAI GPT-4 / 国内可用（DeepSeek/Qwen） | 推理能力要求较高         |
| 运行环境     | Python 3.10+                             | 命令行交互原型           |
| （可选）前端 | Streamlit / Gradio                       | 后续升级为Web界面        |

### 4.2 LangGraph 关键实现需求

python

```
# 必须实现的核心机制
1. State 数据结构定义（明确每个节点读写的字段）
2. interrupt_before=["human_review"] 中断配置
3. 条件边（Conditional Edge）：根据用户选择决定流向
4. 循环回流：reject → 回到 generator（最大循环3次，防止死循环）
5. Checkpointer 持久化：支持中断后恢复会话
```



### 4.3 State 数据结构设计

python

```
from typing import List, Optional, Literal
from typing_extensions import TypedDict

class ChatMessage(TypedDict):
    speaker: Literal["me", "other"]
    content: str

class CandidateReply(TypedDict):
    style: str  # "幽默" | "温暖" | "推拉"
    reply: str
    strategy_explain: str

class SimulationResult(TypedDict):
    best_case: str
    normal_case: str
    worst_case: str
    fallback_suggestion: str

class OverallState(TypedDict):
    # 输入
    chat_history: List[ChatMessage]
    user_persona: Optional[str]
    user_goal: Optional[str]
    
    # Node1 输出
    analysis_report: Optional[dict]
    
    # Node2 输出
    candidates: Optional[List[CandidateReply]]
    
    # Node3 输出（人工审核结果）
    human_decision: Optional[Literal["accept", "modify", "reject"]]
    human_selected_index: Optional[int]
    human_modified_text: Optional[str]
    
    # Node4 输出
    final_reply: Optional[str]
    
    # Node5 输出
    simulation: Optional[SimulationResult]
    
    # 控制字段
    retry_count: int  # 防止无限循环
    error_message: Optional[str]
```



### 4.4 API 接口需求（如用Web）

text

```
POST /api/analyze         # 输入聊天记录，返回分析报告
POST /api/generate        # 生成候选回复
POST /api/review          # 提交人工审核结果（恢复中断）
GET  /api/state/{session_id}  # 查询当前状态
```

## 5. 非功能需求

| 需求       | 描述                                                         |
| :--------- | :----------------------------------------------------------- |
| 响应速度   | 单节点LLM调用不超过20秒，总流程（不含人工等待）< 60秒        |
| 错误处理   | LLM API 调用失败时，能优雅降级并提示用户重试                 |
| 循环保护   | Generator 最多循环3次，若用户连续驳回3次，强制退出并提示"建议手动回复" |
| 会话持久化 | 支持多会话，用户中断后可凭 Session ID 恢复                   |
| 可观测性   | 每个节点执行前后打印 State 关键字段，方便调试                |

## 7. 验收标准

- 能成功运行完整的 5节点 + 1中断 流程
- 人工审核节点能正确暂停并等待输入
- 用户选择"接受"/"修改"/"驳回"后，流程走向正确分支
- 连续驳回3次后，系统能正常退出循环并提示
- 使用 `MemorySaver` 实现中断后状态恢复
- 所有节点的输入输出符合 State 定义
